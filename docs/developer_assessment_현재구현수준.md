# 뷰티 데일리팁 챗봇 — 개발자 관점 현재 구현 수준 판단

**기준**: 회의에서 합의한 5가지 기능(진단 → 상황 진단 → 가이드라인 → 시뮬레이션 → 성분 적합도)  
**코드 수정 없이** 현재 코드베이스만으로 어디까지 가능한지 정리함.

---

## 1. 전체 결론 요약

| 기능 | 회의 합의 번호 | 현재 가능 수준 | 비고 |
|------|----------------|----------------|------|
| 1. 피부 상태 진단 | 1번 | **가능** | 벡터 DB + 웹 검색 + LLM, 톤·형식 구현됨 |
| 2. 전체 상황 진단 | 2번 | **가능** | 진단 결과 이어받아 분석, 생활 패턴 반영 가능 |
| 3. 케어 가이드라인 | 3번 | **가능** | 진단/분석 없으면 노드 내부에서 간이 진단 후 가이드 |
| 4. 변화 시뮬레이션 | 4번 (텍스트 기반) | **가능** | 1주~1년 구간 텍스트 시뮬레이션 구현됨. 단, 진입 경로 제한 있음 |
| 5. 성분별 적합도 | 5번 | **부분 가능** | 벡터 검색·LLM 기반 추천 가능. 구조화된 DB/스키마는 미구축 |

**종합**: **1~4번은 현재 코드만으로 “1차 MVP(진단 → 분석 → 가이드 + 텍스트 시뮬레이션)” 수준까지 실사용 가능.** 5번은 “데이터 기반 정확한 적합도”가 아니라 “벡터+LLM 기반 참고용 추천” 수준까지 가능.

---

## 2. 기능별 상세 판단

### 2-1. 1번 — 사용자 피부 상태 진단 에이전트

**가능 여부: 가능**

- **구현 위치**: `agents/skin_diagnosis_agent.py`, `workflow/nodes.py`의 `diagnosis_node`
- **동작**: 사용자 입력 → 벡터 DB 검색(`피부 타입 진단` + user_input) → 필요 시 웹 검색(`search_with_fallback`) → LLM으로 진단문 생성
- **데이터**: `data/skin_type_cards/` 11종(건성·지성·복합성·수분부족형지성·민감성·장벽손상·각질·피지·트러블·홍조·환경자극민감), `data/피부관리의기초.md`가 벡터 DB에 적재됨(`scripts/init_vector_db.py`)
- **출력 형식**: [피부 타입], [주요 피부 문제], [종합 진단], [격려 메시지] — 회의에서 말한 “친근한 피부관리사 톤”과 맞춤
- **한계**: 벡터/웹 검색에 의존하므로, 답의 일관성·정확도는 프롬프트와 데이터 품질에 좌우됨. “참고용” 디스클로저는 콘텐츠/UI에 별도 추가 필요.

---

### 2-2. 2번 — 전체적인 상황 진단 에이전트

**가능 여부: 가능**

- **구현 위치**: `agents/skin_analysis_agent.py`, `workflow/nodes.py`의 `analysis_node`
- **동작**: 1번 진단 결과(`diagnosis_result`)를 state에서 이어받아, 벡터+웹 검색 후 LLM으로 종합 분석. `lifestyle_info`가 state에 있으면 함께 반영
- **플로우**: 라우터에서 `diagnosis` 선택 시 `diagnosis` → `route_after_diagnosis`에서 성분 키워드 없으면 `analysis` → `analysis_node` 실행
- **출력 형식**: [종합 분석], [생활 패턴 영향], [전체 평가 1~10점], [다음 단계 안내] — 회의에서 요구한 “전체 상황 진단”에 대응
- **한계**: `lifestyle_info`는 현재 state에 넣는 쪽이 구현되어 있지 않음(Streamlit/CLI에서 입력받는 부분 없음). 있으면 그대로 활용 가능하고, 없어도 진단 문장만으로 분석 가능.

---

### 2-3. 3번 — 피부 케어 가이드라인 제공 에이전트

**가능 여부: 가능**

- **구현 위치**: `agents/care_guide_agent.py`, `workflow/nodes.py`의 `care_guide_node`
- **동작**: `diagnosis_result`, `analysis_result`를 state에서 사용. 없으면 노드 안에서 진단 에이전트로 간이 진단 후 가이드 생성. 벡터+웹 검색 후 LLM으로 아침/저녁 루틴·주간 케어·제품 가이드·예산 고려사항 출력
- **진입 경로**: (1) 라우터에서 `care_guide` 의도로 바로 진입, (2) `diagnosis` → `analysis` 후 자동으로 `care_guide` 진입
- **출력 형식**: [아침 루틴], [저녁 루틴], [주간 케어], [제품 선택 가이드], [예산 고려사항] — 회의 합의안의 “가이드라인” 항목과 일치
- **한계**: 없음. 현재 코드만으로 3번 기능은 제공 가능.

---

### 2-4. 4번 — 가이드라인 적용 시 변화 시뮬레이션 (텍스트 기반)

**가능 여부: 가능 (단, 진입 조건 있음)**

- **구현 위치**: `agents/simulation_agent.py`, `workflow/nodes.py`의 `simulation_node`
- **동작**: `care_guide`, `diagnosis_result`를 state에서 받아 벡터+웹 검색 후 LLM으로 1주·1개월·3개월·6개월~1년 구간의 “예상 변화” 텍스트 생성. 이미지/전후 비교 없음 → 회의 합의 “텍스트 기반 4번”과 동일
- **진입 경로**  
  - **A.** 라우터에서 첫 메시지에 “시뮬레이션”이 포함되면 `simulation` 노드로 직행. 이때 `care_guide`/`diagnosis_result`가 없을 수 있음(세션 첫 메시지면 비어 있음). 노드는 빈 값으로도 동작하나, 맥락이 약해짐.  
  - **B.** `diagnosis` → `analysis` → `care_guide`까지 간 뒤, **같은 턴의 user_input에 “시뮬레이션”이 포함된 경우에만** `care_guide` 다음에 `simulation`으로 이동(`graph.py` 조건: `"시뮬레이션" in state.get("user_input", "").lower()`).  
  - 따라서 “진단 한 번 하고 자동으로 분석 → 가이드 → 시뮬레이션”을 한 번에 보려면, 사용자가 **처음부터 “피부 진단해주고 시뮬레이션도 해줘”**처럼 말해야 함. “진단해줘”만 입력하면 4번은 그 턴에서는 실행되지 않음.
- **출력 형식**: [1주일 후], [1개월 후], [3개월 후], [6개월~1년 후], [타임라인 요약] — 회의에서 말한 “텍스트 기반 예상 변화”와 부합
- **한계**: (1) 시각적 전후 비교·이미지 생성은 없음(구현 없음). (2) “가이드 적용 후 시뮬레이션”을 자연스럽게 보여주려면, “가이드 제시 후 다음 메시지에서 시뮬레이션 요청” 시에도 시뮬레이션 노드로 가도록 플로우를 조정하는 게 좋음(현재는 첫 메시지에 “시뮬레이션”이 있어야 같은 턴에서 연결됨).

---

### 2-5. 5번 — 피부 타입/상태 기준 성분별 적합도 에이전트

**가능 여부: 부분 가능 (참고용 수준)**

- **구현 위치**: `agents/ingredient_analysis_agent.py`, `workflow/nodes.py`의 `ingredient_analysis_node`
- **동작**: 벡터 DB(성분 카드) + 웹 검색 + LLM으로 “추천 성분 TOP 5”, “주의 성분”, “성분 조합”, “제품 체크리스트” 등 텍스트 생성. `diagnosis_result`가 state에 있으면 함께 전달됨.
- **데이터**: `data/ingredient_cards/`에 히알루론산·세라마이드·나이아신아마이드 3종만 있음. 그 외 성분은 벡터 검색에 안 걸리면 웹 검색과 LLM 일반 지식에 의존
- **구조화된 DB 미사용**: `get_ingredient_data()`는 TODO로 두고 `return None`. 피부 타입별·상태별 “적합도 점수”를 구조화된 DB/스키마로 관리하는 부분은 없음. 따라서 “성분별 적합도”는 **LLM이 문맥만 보고 생성하는 참고용 답변** 수준
- **진입 경로**: (1) 라우터에서 “성분/원료/추천 성분” 등 의도로 `ingredient_analysis` 직행, (2) `diagnosis` 직후 `route_after_diagnosis`에서 성분 키워드 있으면 `ingredient_analysis`로 분기
- **한계**: (1) 성분 카드가 3개뿐이라 커버리지 제한. (2) RDB/정형 데이터 기반 적합도 매핑이 없어, 회의에서 말한 “검증·업데이트 가능한 성분 적합도”에는 미달. (3) “참고용이며 전문가/제품 설명서 확인 필요” 디스클로저는 콘텐츠/UI에 별도 필요

---

## 3. 워크플로우·인프라 관점

- **LangGraph**: `workflow/graph.py`에서 router → diagnosis / ingredient_analysis / care_guide / simulation 분기, diagnosis → analysis → care_guide 연결, care_guide → (조건부) simulation 연결 구현됨. **에이전트 체이닝은 현재 코드만으로 1~4번까지 구현된 상태.**
- **State**: `workflow/state.py`의 `AgentState`에 `diagnosis_result`, `analysis_result`, `care_guide`, `simulation_result`, `ingredient_analysis_result`, `conversation_history`, `lifestyle_info` 등 필요한 필드 있음. `main.py`에서 이전 턴 결과를 `previous_context`로 넘겨 세션 내 맥락 유지.
- **벡터 DB**: Chroma, OpenAI Embeddings, `init_vector_db.py`로 피부 타입 카드 11종 + 성분 카드 3종 + 교본 1종 적재. 경로는 `config/settings.py`·`.env`의 `VECTOR_DB_PATH`.
- **웹 검색**: DuckDuckGo(`ddgs`)로 벡터 결과 부족 시 보조. `should_search_web` 조건(결과 개수, 길이, “최신” 등 키워드)으로 호출 여부 결정.

## 4. 회의 합의안 대비 체크리스트

| 합의 항목 | 현재 코드 기준 |
|-----------|----------------|
| 1차 MVP: 1번+2번+3번+텍스트 기반 4번 | ✅ 구현됨. 4번은 “첫 메시지에 시뮬레이션 포함” 또는 별도 메시지로 요청 시 동작 |
| 4번 “참고용 예상” 디스클로저 | ❌ 콘텐츠/UI 문구로 아직 없음 |
| 5번 2차, 성분 데이터 출처·스키마 확정 후 도입 | ⚠️ 성분 카드 3종만 있고, 구조화 DB/스키마 없음. 2차에서 데이터·스키마 정한 뒤 확장 필요 |
| 에이전트 체이닝·입출력 포맷 구체화 | ✅ 노드별로 state 입출력이 정해져 있음. 문서화만 보강하면 됨 |

---

## 5. 개발자 관점 최종 정리

- **지금 코드만으로 가능한 것**:  
  - **1~3번**: 피부 진단 → 전체 상황 진단 → 케어 가이드라인까지 한 번에 실행 가능.  
  - **4번**: 텍스트 기반 “가이드 적용 시 예상 변화” 시뮬레이션 가능. 단, 같은 턴에서 진단→가이드 직후 시뮬레이션까지 가려면 첫 메시지에 “시뮬레이션”을 넣어야 함.  
  - **5번**: “벡터+웹+LLM 기반 참고용 성분 추천” 수준까지 가능. “피부 타입/상태별 구조화된 적합도 DB”는 미구현.

- **추가 작업 없이의 한계**:  
  - 4번: 가이드 제시 후 **다음 사용자 메시지**에서 “시뮬레이션 해줘”라고만 해도 시뮬레이션으로 보내는 분기는 현재 그래프에 없음(라우터가 새 의도로 분류할 뿐, “이전 턴 가이드 기준 시뮬레이션”으로 연결되지는 않음).  
  - 5번: 성분 데이터 확대 및 정형 스키마 없이는 “검증 가능한 성분 적합도” 수준에는 미달.

- **실현 가능성 요약**:  
  **1차 MVP(1~3번 + 텍스트 4번)** 는 현재 코드만으로도 **실사용 가능한 수준**이다. 5번은 “참고용 성분 이야기”까지는 가능하고, 회의에서 말한 “2차에서 데이터·스키마 확정 후 도입” 단계는 아직 이 코드만으로는 충족되지 않는다.
